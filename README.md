# moTF Supabase 데이터베이스

이 저장소는 이용자 웹과 사장님·운영팀 웹이 함께 사용하는 **유일한 DB 변경 원본**입니다.
다른 저장소에 SQL 복사본을 두지 않습니다.

## 적용 원칙

1. SQL은 파일 번호 순서대로 실행합니다.
2. 이미 실행한 파일은 수정하거나 다시 실행하지 않습니다.
3. 변경은 항상 다음 번호의 새 파일로 추가합니다.
4. SQL 실행 전 Supabase 백업과 대상 프로젝트를 확인합니다.
5. 비밀번호, `service_role`, 결제 Secret은 저장소에 기록하지 않습니다.

## 마이그레이션

- `01`~`08`: 회원·업장·권한·예약·채팅·운영 콘텐츠 기반
- `09`~`12`: 이용자/파트너 역할 분리와 가입·승인 흐름
- `13`~`15`: 공개 카탈로그·이미지·실제 예약/주문
- `16`: 실시간 업장 채팅
- `17`: 채팅 읽음과 문의·분쟁 관리
- `18`: 운영 안정화와 가격 검증
- `19`: 잘못 저장된 파트너 역할 복구와 승인 함수 보강
- `20`: 파트너 프로필과 업장 승인 상태 동기화
- `21`: 결제 대기 원장과 승인 후 예약·주문 생성
- `22`: 업장 주소를 네이버 지도 마커용 좌표로 저장
- `23`: 주소 검색 API에서 선택한 우편번호와 상세주소 저장
- `24`: 선결제 후 거절 시 환불 필요 상태 저장
- `25`: 결제 완료 기록이 있는 요청만 환불 필요 상태로 표시
- `26`: 포트원 KG이니시스 가상계좌 발급·입금 완료 흐름 저장

`18_production_hardening.sql`은 다음을 고칩니다.

- 로그인 이용자의 공개 상품 조회
- DB 원본 가격으로 예약·주문 금액 계산
- 다른 업장 상품과 잘못된 수량 차단
- 예약 직접 삽입 차단과 검증 RPC 제공
- 상품 수정 시 기존 상품 ID 유지

## 현재 적용 기준

운영 DB는 `01`~`26`까지 순서대로 적용하는 것을 기준으로 합니다.
`22`와 `23`은 사장님 앱의 주소 검색, 주소 위치 확인, 이용자 지도 마커 표시를 위해
필요합니다. `24`는 선결제 후 사장님 또는 운영팀이 요청을 거절했을 때 환불 처리가
처리할 수 있는 `refund_status`를 남깁니다. `25`는 결제 완료 기록이 없는 과거 테스트
데이터가 환불 예정으로 잘못 표시되지 않도록 보정합니다. `26`은 포트원
KG이니시스 가상계좌를 사용하도록 결제 원장 상태를 확장합니다.

신규 환경이나 백업 복구 환경에서는 다음 순서로 진행합니다.

1. Supabase SQL Editor에서 마지막 적용 번호를 확인합니다.
2. 미적용 SQL을 번호 순서대로 실행합니다.
3. 이용자 웹과 사장님 웹 수정본을 배포합니다.
4. `docs/production-hardening-checklist.md`로 역할별 테스트를 진행합니다.

SQL 실행 여부를 확인하지 않고 같은 파일을 반복 실행하지 마세요. 적용 후에는 이 README와
현황 문서의 적용 번호를 함께 갱신합니다.
