# moTF 데이터베이스

이 저장소는 이용자 웹과 사장님·관리자 웹이 함께 사용하는 Supabase 데이터베이스의 유일한 SQL 원본입니다.
앱 저장소에는 SQL 사본을 두지 않습니다.

## 기본 원칙

1. SQL은 `supabase` 폴더의 파일명 순서대로 실행합니다.
2. 이미 성공적으로 실행한 SQL은 다시 실행하지 않습니다.
3. 변경 사항은 기존 파일 수정이 아니라 다음 번호의 새 SQL로 추가합니다.
4. 실행 전 Supabase 백업과 대상 프로젝트를 확인합니다.
5. 비밀번호, `service_role` 키, 결제·알림 API Secret은 저장소에 기록하지 않습니다.

## 파일명 규칙

- 일반 마이그레이션: `01_description.sql`
- 같은 단계에서 순서가 필요한 경우: `26A_description.sql`, `26B_description.sql`
- 롤백 및 긴급 복구 스크립트: `docs/archive`에 보관

`26A` 다음에는 `26B`, `29A` 다음에는 `29B`, `42A` 다음에는 `42B`를 실행합니다.
28번은 이후 30번의 공실·정산 통합 마이그레이션으로 대체되어 현재 파일이 없습니다.

## 기능 구간

- `01`~`20`: 회원, 권한, 업장, 상품, 예약, 주문, 채팅
- `21`~`29B`: 결제 요청, 포트원 가상계좌, 환불 상태, 주소·좌표
- `30`~`36`: 공실 방막기, 입금 대기, 정산, 자동 만료, 자동 환불
- `37`~`39`: 보안 경계, 카카오 로그인 프로필, 회원 탈퇴
- `40`~`43`: 알림 발송 대기열, 이벤트 연결, 즉시 발송 및 전체 알림 범위

## 새 환경 적용

1. Supabase SQL Editor에서 대상 프로젝트를 확인합니다.
2. `supabase` 폴더 파일을 이름순으로 하나씩 실행합니다.
3. 오류가 발생하면 다음 파일로 넘어가지 말고 실행을 중단합니다.
4. 이용자 웹과 사장님 웹에서 권한별 핵심 흐름을 확인합니다.
5. 결제와 알림 Secret은 각 배포 프로젝트의 환경변수에만 등록합니다.

## 현재 운영상 주의

- 포트원 결제와 환불 함수는 여러 보완 마이그레이션을 거쳤으므로 중간 파일을 생략하면 안 됩니다.
- 입금 대기 상태도 객실 재고를 임시 점유하며, 만료되면 방막기가 해제됩니다.
- 알림 시스템은 딜러사 연동 전까지 `mock` 발송 상태를 사용할 수 있습니다.
- 롤백 파일은 정상 설치 과정에서 실행하지 않습니다.

세부 운영 점검은 `docs/production-hardening-checklist.md`와 알림 문서를 함께 확인합니다.
